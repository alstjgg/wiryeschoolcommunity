# 위례인생학교 업무 도우미 — 개발 기획서

> **문서 버전**: 4.1  
> **최종 수정**: 2026-03-07  
> **프로젝트명**: 위례인생학교 업무 도우미 (Wirye School Assistant)  
> **대상 독자**: 개발자, 프로젝트 관리자

---

## 1. 프로젝트 개요

### 1.1 목적과 가치

위례인생학교는 성인 대상 평생교육 프로그램으로, 연 4학기(1월/4월/7월/10월)에 걸쳐 다양한 강좌를 운영한다. 현재 2~4명의 관리자가 수강신청 접수, 입금 확인, 강의계획서 검토, 출석부 생성, 보고서 작성 등의 업무를 수작업으로 처리하고 있으며, 학기 시작 전 2~3주간 업무 부하가 집중된다.

이 프로젝트는 LLM과 Google Sheets API를 결합하여 관리자의 반복 업무를 자동화하되, **완벽한 자동화가 아니라 "수시간 → 수분"으로 단축하는 AI 보조 도구**를 목표로 한다. AI가 90%를 처리하고, 나머지 10%를 관리자가 빠르게 확인·수정하는 구조이다.

### 1.2 핵심 설계 원칙

- **프로토타입 우선**: 최소한의 뼈대(챗봇 + 배포)부터 만들고 점진적으로 기능을 추가한다.
- **완벽보다 속도**: AI 출력의 100% 정확성을 추구하지 않고, 관리자가 검증 가능한 구조화된 결과를 빠르게 제공한다.
- **LLM은 예외 처리 전담**: 코드로 처리 가능한 규칙 기반 로직은 코드로, LLM은 비정형 데이터 해석에만 사용하여 비용과 정확도를 최적화한다.
- **폴더/파일 명명 규칙이 곧 API**: 예측 가능한 파일 경로로 탐색 없이 직접 접근한다.

### 1.3 범위

**포함 (In-scope)**

- AI 기반 대화형 업무 도우미 (LangChain + Chainlit)
- Google Drive/Sheets 기반 데이터 저장/조회/수정
- 6개 핵심 업무 자동화: 입금 대조, 계획서 검토, 출석부 생성, 출석 체크(OCR), 보고서 생성, 질의응답
- 클라우드 배포 (PaaS)

**제외 (Out-of-scope)**

- 배움숲 포털과의 직접 API 연동 (API 미제공)
- 수강생 대면 서비스 (관리자 전용 시스템)
- 모바일 전용 앱 개발
- 실시간 채팅/메시징, 일정 기반 능동적 알림 (추후 확장)

### 1.4 대상 사용자

위례인생학교 관리자 2~4명. IT 전문 인력이 아니며, 웹 브라우저에서 채팅 형태로 업무를 요청한다. 학기 시작 전 피크 시즌(연 4회, 각 2~3주)에 집중 사용하고, 평상시에는 간헐적으로 조회 용도로 사용한다.

## 2. 기술 스택

### 2.1 구성요소

| 구성요소 | 기술 | 용도 | 선택 근거 |
|---------|------|------|----------|
| LLM 프레임워크 | LangChain | LLM 파이프라인, 의도 분류, Tool 호출 | Python 네이티브, 커스텀 로직 제한 없음, 모델 교체 용이 |
| 채팅 UI | Chainlit | 대화형 웹 인터페이스, 버튼 UI | Python 통합, Conversation Starter 버튼 네이티브 지원 |
| LLM | Claude API (Anthropic) | 의도 파악, 텍스트 처리, OCR (Vision) | 한국어 성능 우수, Vision 지원 |
| 데이터 저장 | Google Drive / Sheets API | 모든 업무 데이터 저장/관리 | 관리자가 직접 확인/수정 가능, 별도 DB 불필요 |
| Google API 인증 | Service Account + Domain-wide Delegation | Drive/Sheets 읽기/쓰기 | SA가 Workspace 사용자 대리로 파일 생성 가능 |
| 배포 (Phase 0) | Fly.io | 프로토타입 빠른 배포 | CLI 한 줄 배포, 무료 티어 |
| 배포 (Phase 1~) | Railway | 운영 환경 | Git push 자동 배포, HTTPS/도메인 자동, 관리 부담 최소 |
| 프로그래밍 언어 | Python 3.10+ | 전체 앱 | LangChain/Chainlit 네이티브 |

### 2.2 기술 선택 근거

**LangChain + Chainlit을 선택한 이유**: Google Drive/Sheets API를 직접 호출하여 시트 단위 읽기/쓰기, 파일 업로드 등을 자유롭게 제어할 수 있다. 입금 매칭처럼 복잡한 로직을 Python 코드로 제한 없이 구현 가능하고, 외부 라이브러리(openpyxl, pandas 등)를 자유롭게 사용할 수 있다. Chainlit은 Conversation Starter 버튼을 네이티브로 지원하여 관리자가 주요 작업에 바로 접근할 수 있다.

**Railway를 선택한 이유**: Git push만으로 자동 배포되어 운영 부담이 최소화된다. HTTPS와 커스텀 도메인이 자동 제공되고, 서버 관리(OS 업데이트, Docker 관리 등)가 불필요하다. WebSocket(Chainlit 필수)을 지원하며, 월 $5~7 수준으로 예산 내에서 운영 가능하다.

**Context Injection을 선택한 이유 (RAG 대체)**: 비즈니스 컨텍스트(학기 구조, 환불 규칙, 회원 제도 등)가 2~3페이지 분량으로 벡터DB+임베딩+검색 파이프라인을 구축할 만큼의 데이터 양이 아니다. 시스템 프롬프트에 직접 주입하는 것으로 충분하며, 인프라 복잡도를 크게 줄인다.

**Google Sheets를 데이터 저장소로 선택한 이유**: 별도 DB 구축 없이 관리자가 직접 데이터를 확인/수정할 수 있다. 서버가 stateless가 되어 재시작/재배포 부담이 없다. Sheets API가 시트/셀 단위로 읽기/쓰기를 지원하므로, 1파일 다중시트 구조도 프로그래밍으로 자유롭게 처리 가능하다.

### 2.3 시스템 아키텍처

```
┌──────────────────┐       ┌───────────────────────────────────────┐
│  관리자 브라우저   │       │         Railway (PaaS)                 │
│                  │ HTTPS │                                       │
│  ai.wirye        ├──────►│  ┌─────────────────────────────────┐  │
│  schoolcommunity │       │  │  Python App                     │  │
│  .com            │       │  │                                 │  │
│                  │       │  │  Chainlit ── LangChain ── 로직  │  │
│                  │       │  │  (채팅 UI)   (LLM 파이프라인)     │  │
│                  │       │  │                                 │  │
│                  │       │  │  Context     Google API          │  │
│                  │       │  │  Injection   (Drive/Sheets)      │  │

│                  │       └────────────────┬──────────────────────┘
└──────────────────┘                       │
                            ┌──────────────┴──────────────┐
                            │  ┌────────┐  ┌───────────┐  │
                            │  │ Google │  │ Claude    │  │
                            │  │ Drive/ │  │ API       │  │

                            └─────────────────────────────┘
```

핵심 특징: DB 없음(Google Sheets가 저장소), 서버 stateless, Git push 자동 배포.

## 3. 비즈니스 컨텍스트

### 3.1 학기 구조

연 4학기, 각 학기 3개월, 강좌당 최대 12회차.

| 학기 | 시작월 | 기간 | 개강 기준 |
|:---:|:-----:|:----:|----------|
| 1학기 | 1월 | 1~3월 | 해당 월 첫 주 |
| 2학기 | 4월 | 4~6월 | 동일 |
| 3학기 | 7월 | 7~9월 | 동일 |
| 4학기 | 10월 | 10~12월 | 동일 |

### 3.2 회원 등급 체계

3단계 등급 구조. 승급/강등이 학기 흐름에 따라 자동 또는 입금 기반으로 전환된다.

```
[비회원] ──가입비──→ [회원] ──수강료 입금──→ [준회원] ──종강──→ [회원]
                       │
                       └──정회원비──→ [정회원] ──만료학기 경과──→ [회원]
```

| 등급 | 승급 조건 | 수강 신청 | 수강료 | 유효기간 | 만료 시 |
|------|---------|:--------:|:------:|---------|--------|
| 회원 | 가입비 납부 | 가능 | 필요 | 영속 | — |
| 준회원 | 수강료 입금 완료 (1과목 이상) | 등록 완료 | 납부 완료 | 해당 학기 종강까지 | 회원으로 강등 |
| 정회원 | 정회원비 납부 | 가능 | **면제** | ~후년 1학기 | 회원으로 강등 |

비회원은 시스템 외부 상태이며, 가입 완료 시 회원관리 시트에 추가된다.

**특수 케이스**:
- 강사: 자동으로 정회원 자격 부여 (시작/만료 기준은 추후 확정)
- 사무처 직원: 입사 시 정회원, 퇴사 시 해당 학기까지 유지 → 다음 학기 만료

### 3.3 비용 구조

| 항목 | 기본 금액 | 비고 |
|------|:--------:|------|
| 가입비 | 1만원 | 변동 가능 |
| 수강료 | 2만원 (전 과목 동일) | 변동 가능 |
| 정회원비 | 12만원 | 매년 초(약 2월) 결정, 변동 |

모든 비용은 변동 가능하므로 코드 하드코딩이 아닌 설정값으로 관리한다.

### 3.4 환불

- 환불 마감: 개강 후 7일, 이후 환불 불가

### 3.5 사용자 식별

- **User ID**: `이름+핸드폰번호뒷자리` (예: `박민서6804`)
- 동명이인 구분 목적
- 마스터 시트 및 출석부의 기본 키

### 3.6 주요 업무 일정

| 시점 | 업무 |
|-----|------|
| 학기 시작 2달 전 | 강사에게 강의 계획서 요청 |
| 학기 시작 1달 전 | 계획서 검토, 배움숲/카페 홍보 게시 |
| 전월 말일까지 | 수강신청 접수 마감 |
| 개강 첫 주 전 토요일 | 강사에게 수강자 목록(출석부) 전달 |
| 개강 1주 후 | 환불 마감 |
| 학기 종료 후 | 출석 체크(OCR), 운영 보고서 작성 |

## 4. 데이터 구조

> **참고**: 이 섹션의 데이터 구조는 개발 편의를 위한 **제안(안)**이다. 기획/운영 측 요구에 따라 컬럼, 시트 구성, 파일 구조 모두 변경 가능하며, 개발 과정에서 실제 운영 상황에 맞춰 조정한다.

### 4.1 Google Drive 폴더 구조

```
위례인생학교 자료실/
├── 회원관리.xlsx                    ← 전체 회원 마스터 (영속, 학기 무관)
├── 수강기록.xlsx                    ← 전체 수강 이력 (영속, 학기 무관)
└── 학사운영(연도별)/
    └── {연도}/
        └── {YYYY}_{N}학기/
            ├── 수강생.xlsx           ← 해당 학기 수강 신청 + 입금 관리
            ├── 출석부/
            │   └── {강좌명}_출석부.xlsx
            ├── 입금내역/
            │   └── {YYYY}_{MM}_{DD}_입금내역.xlsx
            └── 강의계획서/
                ├── 초안/
                └── 검토완료/
```

루트 레벨에 영속 데이터(회원관리, 수강기록), 학기 폴더에 학기별 데이터(수강생, 출석부, 입금내역). 파일명에 연도/학기/과목이 포함되어 AI가 경로를 직접 구성할 수 있다.

**파일 구조 옵션**: Google Sheets API는 시트/셀 단위로 직접 읽기/쓰기를 지원하므로, 1파일 다중시트도 가능하고 개별 파일도 가능하다. 개발 중 운영 편의성과 성능을 고려하여 확정한다.

### 4.2 테이블 관계

```
[회원관리] ←──집계── [수강기록]
 (루트, 영속)          (루트, 영속)
     │                    ▲
     │ 이름ID             │ 학기 종료 시 추가
     ▼                    │
[수강생]              [출석부]
 (학기 폴더)           (학기 폴더)
```

- **회원관리**: 전체 회원 마스터 + type/active 관리. 학기를 넘어 영속.
- **수강기록**: 전체 수강 이력 (학기×과목×출석률). 회원관리의 수강count/출석률 산출 원본.
- **수강생**: 해당 학기의 수강 신청 + 입금 관리. 학기마다 새로 생성.
- **출석부**: 해당 학기 강좌별 출석 체크. 학기마다 새로 생성.

### 4.3 회원관리 (루트, 영속)

전체 회원(과거+현재)을 관리하는 마스터 테이블. 신규 가입 시 행 추가, 학기가 넘어가도 유지.

| 컬럼 | 설명 | 비고 |
|------|------|------|
| 이름ID | 이름+뒷자리 (예: 박민서6804) | PK |
| 이름 | | |
| 성별 | | |
| 전화번호 | 010-XXXX-XXXX | |
| 주소 | | |
| 나이 | | |
| 등급 | 회원 / 준회원 / 정회원 | 승급/강등 자동 관리 |
| 가입날짜 | | |
| 시작학기 | 가입한 학기 (예: 2026_1학기) | |
| 만료학기 | 정회원: 후년 1학기, 회원/준회원: 없음 | |
| 수강 count | 누적 수강 횟수 | 수강기록에서 집계 |
| 출석률 (누적) | 전체 평균 출석률 | 수강기록에서 집계 |
| 마지막수강학기 | | 수강기록에서 집계 |
| active | Y/N | 매 1학기(1월) 일괄 업데이트 |

**등급 관리 로직**:
- **회원**: 가입비 납부 완료. 수강 신청 가능. 영속(만료 없음).
- **준회원**: 수강료 입금 확인 시 회원에서 승급 (1과목 이상). 종강 후 회원으로 자동 강등.
- **정회원**: 정회원비 납부 시 회원에서 승급. 수강료 면제. 만료학기(후년 1학기) 경과 시 회원으로 자동 강등.
- 강사: 자동 정회원 (시작/만료 기준 추후 확정)
- 사무처 직원: 입사 시 정회원, 퇴사 시 해당 학기까지 유지 → 다음 학기 만료

**자동 등급 전환 시점**:
- 학기 시작 시: 정회원 중 현재 학기 > 만료학기 → 등급 = "회원"
- 입금 대조 시: 수강료 입금 확인 → 등록상태 "정상등록" + 회원이면 등급 = "준회원"
- 종강 시: 준회원 → 등급 = "회원"

### 4.4 수강기록 (루트, 영속)

전체 수강 이력을 관리하는 테이블. 학기 종료 시 출석부 데이터를 바탕으로 행을 추가한다.

| 컬럼 | 설명 |
|------|------|
| 이름ID | 회원관리와 연결 |
| 학기 | 2026_1학기 등 |
| 과목명 | |
| 출석률 | 해당 강좌의 출석률 |

학기 종료 시 흐름: 출석부(학기 폴더) → 출석률 확정 → 수강기록에 행 추가 → 회원관리의 수강count/출석률/마지막수강학기 갱신.

### 4.5 수강생 (학기 폴더, 학기별)

해당 학기의 수강 신청자와 입금 상태를 관리하는 테이블. 학기마다 새로 생성. 다과목 수강자는 이름ID+과목명이 복합 PK로, 과목마다 행이 추가된다.

| 컬럼 | 설명 | 입력 주체 |
|------|------|----------|
| 이름ID | 회원관리와 연결 | 초기 입력 |
| 과목명 | | 초기 입력 |
| 입금시간 | 매칭된 입금 건의 거래일시 | **AI 자동** |
| 입금자명(적요) | 매칭된 입금 건의 은행 적요 (원본) | **AI 자동** |
| 비고 | | |
| 입금현황 | ✅정상 / 🔶확인필요 / ❌미입금 / 💎면제 | **AI 자동** |
| 등록상태 | 정상등록 / 대기 / 취소 | 관리자 확인 후 최종 결정 |

입금시간~입금현황 컬럼은 AI가 입금 대조 시 자동으로 채운다. 정회원은 수강료 면제이므로 입금현황 = "💎면제", 등록상태 = "정상등록"으로 자동 처리된다. 관리자가 수강생 시트 한 화면에서 매칭 결과 전체를 확인·검증할 수 있는 구조.

### 4.6 출석부 (학기 폴더, 강좌별)

| 컬럼 | 설명 |
|------|------|
| ID | 이름+뒷자리 |
| 1회차 ~ 12회차 | 출석/결석/지각 등 (12회차 일괄 생성, 미만 회차도 12컬럼) |
| 출석률 | 출석수 / 회차수 × 100 (수식 삽입) |

등록상태가 정상인 수강자만 포함. 프린트를 위한 PDF 변환 필요.

### 4.7 입금내역 (은행 다운로드 원본)

은행에서 다운받은 엑셀을 학기 폴더 내 입금내역/ 에 그대로 업로드한다.

| 컬럼 | 설명 | 비고 |
|------|------|------|
| 거래일시 | 2025-12-31 23:42:54 | |
| 적요 | 비정형 입금자 입력 텍스트 | 이름+강좌 혼합, 형식 자유 |
| (비고) | 은행 비고/메모 | 정규화된 강좌명 포함, 매칭에 핵심 |
| 의뢰인/수취인 | 은행 계좌 명의자 | 대리입금 시 적요와 다름 |
| 입금 | 금액 | |
| 출금 | | |
| 거래후잔액 | | |
| 구분 | 대체/타행이체 | |
| 거래점 | 은행명(계좌뒷자리) | |

헤더가 7행에 위치하고, 1~6행은 계좌 메타정보(예금주, 계좌번호, 조회기간), 마지막 행은 합계이다. 파싱 시 이를 고려해야 한다.

### 4.8 입금자명 패턴 (실제 데이터 기반)

은행 적요 컬럼은 입금자가 자유롭게 입력하는 텍스트로, **형식이 통일되어 있지 않다**. 실제 확인된 주요 패턴:

| 패턴 | 예시 | 빈도 |
|------|------|------|
| 이름+강좌(구분자 없음) | `김기춘경제뉴스로기초` | 매우 많음 |
| 이름_강좌 | `류동원_요들` | 일부 |
| 이름 강좌(공백) | `성윤경 경제뉴스심화` | 일부 |
| 강좌+이름 (역순) | `경제기초조윤정` | 일부 |
| 강좌(이름) | `황용섭(경제)` | 일부 |
| 이름-강좌, 이름/강좌, 이름.강좌 | `김혜경-심리상담`, `조현진/금융과경제` | 일부 |
| 이름만 | `최정숙`, `심필례` | 일부 |
| 가입비 | `이지희 가입비`, `정혜숙준회원신규회비` | 가입비 건 |
| 연회원 | `올영어김은희정회원` 10만원 | 연회원 건 |
| 합산 입금 | `오예은+경제(기초 심화)` 4만원 | 다과목 합산 |
| 대리 입금 | 적요 `김종렬` / 의뢰인 `김옥자` (가족 대리) | 간헐 |
| 간편결제 | 의뢰인이 `(주)카카오페이`, `(주)비바리퍼블리카` | 간헐 |

비고 컬럼이 상대적으로 정규화된 강좌명을 포함하므로, 매칭 시 강좌명은 비고 컬럼 우선, 이름은 의뢰인 컬럼 우선으로 활용한다.

### 5.2 입금 대조 (1순위)

AI 효율화의 핵심 작업. 코드 기반 규칙 매칭(80~90%) + LLM 기반 예외 처리(10~20%) 2단계 구조.

**입력**: 연도, 학기 (자동 추론 + 사용자 확인)

**처리 개요**:
1. 사용자가 "입금 대조" 버튼 클릭 → 연도/학기 확인
2. Google Sheets API로 입금내역 + 수강생 시트 + 회원관리 시트 읽기
3. **정회원 자동 처리**: 회원관리에서 등급이 "정회원"인 수강생은 입금현황 = "💎면제", 등록상태 = "정상등록"으로 자동 설정 (입금 매칭 대상에서 제외)
4. **코드 기반 매칭**: 의뢰인 이름으로 수강생 이름 매칭 → 비고 컬럼으로 강좌 매칭 → 금액 분류(1만=가입비, 2만=수강료, 3만=합산 등) → 취소/대기 키워드 감지
5. **LLM 처리** (매칭 실패/애매한 건만): 비정형 적요 텍스트를 해석하여 가장 가능성 높은 매칭 후보 제시
6. 결과를 관리자에게 요약 제시 → 확인 후 수강생 시트에 write-back
7. **등급 승급**: 입금 확인된 회원 중 등급이 "회원"인 사람 → 회원관리의 등급을 "준회원"으로 승급

**출력**: 수강생 시트의 입금시간/입금자명(적요)/입금현황 컬럼 자동 채움 + 회원관리 등급 승급

**입금 상태 코드**:

| 코드 | 이모지 | 의미 |
|-----|:-----:|------|
| confirmed | ✅ | 정상 매칭 |
| needs_review | 🔶 | 확인 필요 (LLM 추정, 동명이인, 금액 불일치 등) |
| name_mismatch | ⚠️ | 이름 불일치 (대리 입금 추정) |
| not_paid | ❌ | 미입금 |
| duplicate | 🔄 | 중복 입금 |

### 5.3 출석부 생성 (1순위)

**입력**: 연도, 학기, 전체 or 특정 강좌

**처리 개요**:
1. 수강생 시트에서 등록상태 "정상등록" 수강자 필터링
2. 강좌별 그룹핑 → 중간 확인 ("금융과 경제: 17명, ...")
3. 강좌별 출석부 생성 (ID, 이름, 1~12회차, 출석률 수식)
4. PDF 변환 (프린트용)
5. Google Drive 출석부 폴더에 업로드

### 5.4 출석 체크 — OCR (2순위)

종강 후 강좌별 1회 진행. 실물 출석부(12회차 전체)를 사진으로 촬영하여 디지털화.

**처리 개요**:
1. 출석부 사진 업로드
2. Claude Vision으로 이름별 × 회차별 출석/결석 판독
3. 기존 출석부 시트의 ID와 매칭
4. 결과 확인 후 출석부 시트 업데이트 + 출석률 재계산

### 5.5 계획서 검토 (2순위)

**지원 형식**: PDF만 (HWP 미지원)

**처리 우선순위**:
- 1순위: 오탈자 수정, 말투 수정, 배움숲 업로드용 카테고리별 복붙 멘트 생성 (채팅으로 전달)
- 후순위(이번 미구현): 내용 보강, 용어 보완, 네이버 카페 멘트 생성

**출력물**: '강의 계획서 정리' 파일 — 강좌별 시트, A열(템플릿 항목: 강좌명/학기/모집대상/강사명 등), B열(복붙용 멘트)

### 5.6 보고서 생성 (3순위)

학기 종료 후 운영 현황 보고서 자동 생성. 상세 설계는 추후 확정.

### 5.7 질의 응답

비즈니스 규칙(학기 일정, 환불 규칙, 회원 제도 등)에 대한 자연어 Q&A. 시스템 프롬프트에 비즈니스 컨텍스트를 직접 주입하여 처리한다.

### 6.1 Context Injection

비즈니스 컨텍스트 양이 소량(2~3페이지)이므로, 시스템 프롬프트에 직접 주입한다.

```python
# 정적 컨텍스트 — 코드에 포함, 비즈니스 규칙 변경 시 수정
BUSINESS_CONTEXT = {
    "학기구조": "연 4학기 (1월/4월/7월/10월), 각 3개월, 최대 12회차...",
    "회원제도": "일반회원 가입비 1만원, 연회원 12만원...",
    "수강료": "기본 2만원, 환불 마감 개강 후 7일...",
    "입금패턴": "입금자명 형식 비정형..."
}

# 동적 컨텍스트 — 런타임에 Google Sheets에서 로드
SEMESTER_CONTEXT = {
    "회원목록": [],     # Sheets에서 로드
    "강좌목록": [],     # Sheets에서 로드
    "입금내역": [],     # Sheets에서 로드
}

# 작업별 필요한 컨텍스트만 선택 주입 → 토큰 절약
def build_prompt(task_type, data):
    base = BUSINESS_CONTEXT["학기구조"] + BUSINESS_CONTEXT["회원제도"]
    if task_type == "입금대조":
        return base + format_payment_context(data)
    elif task_type == "질의응답":
        return base + BUSINESS_CONTEXT["수강료"]
```

### 6.2 Google API 인증 (Domain-wide Delegation)

Service Account에 Domain-wide Delegation을 설정하면, SA가 Workspace 사용자(`wirye@wiryeschoolcomunity.com`) 대리로 파일을 생성/수정할 수 있다. 파일 소유자가 Workspace 사용자가 되므로 스토리지 쿼터 문제가 발생하지 않는다.

```python
from google.oauth2 import service_account

credentials = service_account.Credentials.from_service_account_file(
    'sa-key.json',
    scopes=[
        'https://www.googleapis.com/auth/drive',
        'https://www.googleapis.com/auth/spreadsheets'
    ],
    subject='wirye@wiryeschoolcomunity.com'
)
```

**설정 절차**:
1. GCP Console → IAM & Admin → Service Accounts → 해당 SA에 Domain-wide Delegation 활성화 → Client ID 복사
2. Google Workspace Admin Console → Security → API controls → Manage Domain Wide Delegation → Client ID + OAuth Scopes(`drive`, `spreadsheets`) 등록

## 10. 잠재적 위험 및 완화 전략

| 위험 | 영향 | 완화 전략 |
|-----|------|----------|
| Railway 서비스 장애/가격 변경 | 일시적 중단 또는 비용 증가 | 코드가 GitHub에 있으므로 Fly.io/Render 등으로 즉시 이관 가능 |

| 은행 엑셀 형식 변경 | 파싱 실패 | 유연한 파싱 로직 + 형식 변경 감지 시 에러 안내 |
| AI 매칭 정확도 부족 | 관리자 신뢰 저하 | 🔶 확인 필요 상태로 관리자 최종 검증 보장, AI는 보조 도구 역할 |
| 개발자 부재 시 유지보수 | 관리 공백 | PaaS 배포로 서버 관리 불필요, 상세 운영 가이드 제공 |
