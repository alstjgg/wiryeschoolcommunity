## 1. 프로젝트 개요

### 1.1 목적

위례인생학교 관리자의 반복 업무를 지원하는 AI 기반 업무 도우미 개발

### 1.2 핵심 기능

| 기능 | 설명 | AI 필요 | Phase |
|-----|------|:------:|:-----:|
| 계획서 검토 | 강사 제출 계획서 검토, 말투/용어 수정, 게시글 생성 | ✅ | 2 |
| 입금 대조 | 은행 입금 내역과 수강신청자 매칭 | ❌ | 2 |
| 수강자 목록 생성 | 강사별 수강자 목록 생성 | ❌ | 2 |
| 보고서 생성 | 학기별/연도별 운영 현황 보고서 | △ | 2 |
| 출석부 관리 | 종이 출석부 이미지 → 데이터 변환 | ✅ | 2 |
| 능동적 업무 알림 | 일정 기반 업무 안내 | ❌ | 3 |

### 1.3 추가 컨텍스트 기능 (Knowledge Base)

| 기능 | 설명 | 구현 방식 | Phase |
|-----|------|----------|:-----:|
| 환불 규칙 안내 | "환불 가능한가요?" 질문에 응답 | Knowledge Base | 1 |
| 블랙리스트 분석 | 취소율/출석률 기반 관리 대상 분석 | Knowledge Base + Agent 분석 | 1 |

### 1.4 설계 원칙

- **AI Agent 중심**: 사용자와의 모든 상호작용은 AI Agent를 통해 이루어짐
- **점진적 시스템화**: 반복 작업은 워크플로우로 자동화하여 토큰 비용 최적화
- **단순한 배포**: Docker 기반으로 설치/관리 용이

## 3. 기술 스택

### 3.1 핵심 구성요소

| 구성요소 | 기술 | 용도 | 버전 |
|---------|------|------|------|
| AI Platform | Dify | Agent 호스팅, 대화 UI | latest |
| LLM | Claude API (Anthropic) | 의도 파악, 텍스트 처리, 응답 생성 | claude-3-5-sonnet |
| 자동화 | n8n | 워크플로우 실행 (Phase 2) | latest |
| 파일 저장소 | Google Drive | 모든 업무 데이터 저장 | - |
| 컨테이너 | Docker / Docker Compose | 배포 및 운영 | 24.x |

### 3.2 Dify 구성

```yaml
# Dify 앱 구성
App Type: Chatbot (Agent)
Model: Claude 3.5 Sonnet

Features:
  - Conversation Opener: 4개 버튼 (Phase 2)
  - Knowledge Base: 업무 컨텍스트 문서
  - Tools: HTTP Request (n8n 연동)
```

### 3.3 n8n 워크플로우 (Phase 2)

| 워크플로우 | 트리거 | 주요 노드 |
|-----------|--------|----------|
| 입금 대조 | Webhook | Google Drive → 파싱 → 매칭 로직 → 저장 → 응답 |
| 계획서 검토 | Webhook | Google Drive → 추출 → Claude AI → 저장 → 응답 |
| 수강자 목록 | Webhook | Google Drive → 필터링 → 포맷팅 → 응답 |
| 보고서 생성 | Webhook | Google Drive → 집계 → 포맷팅 → 저장 → 응답 |
| 출석부 관리 | Webhook | 이미지 수신 → Claude Vision OCR → 데이터 추출 → 저장 → 응답 |

### 3.4 n8n 워크플로우 (Phase 3)

| 워크플로우 | 트리거 | 주요 노드 |
|-----------|--------|----------|
| 업무 알림 | Scheduler (매일) | 날짜 체크 → 업무 판단 → Dify Agent에 알림 메시지 전달 |

## 4. 개발 Phase

### 4.1 Phase 1: 기반 구축 + 범용 Agent

**목표**: 업무 컨텍스트를 이해하고 Google Drive와 상호작용 가능한 Agent

**기간**: ~2026년 2월 말 (2학기 시작 전)

**범위**:
- [ ] Dify 환경 구축 (Cloud → Docker)
- [ ] Agent 프롬프트 작성 (비즈니스 컨텍스트)
- [ ] Knowledge Base 구축 (환불 규칙, 블랙리스트 기준 포함)
- [ ] Google Drive 연동 (읽기/쓰기)
- [ ] 기본 대화 테스트
- [ ] Docker 배포 패키지 생성

**산출물**:
- Docker로 실행 가능한 Conversational AI Agent
- 사용자 매뉴얼 (관리자용)

**Agent 기능**:
```
✅ 업무 관련 질문 응답
✅ Google Drive 파일 조회
✅ 데이터 조회/검색
✅ 간단한 데이터 작성
✅ 환불 규칙 안내
✅ 블랙리스트 기준 기반 분석 (요청 시)
❌ 자동화된 워크플로우 (Phase 2)
```

### 4.2 Phase 2: 업무 시스템화

**목표**: 5가지 핵심 기능을 n8n 워크플로우로 자동화

**기간**: ~2026년 6월 말 (3학기 시작 전)

**범위**:
- [ ] n8n 환경 구축
- [ ] 워크플로우 5개 개발
  - [ ] 입금 대조
  - [ ] 계획서 검토
  - [ ] 수강자 목록 생성
  - [ ] 보고서 생성
  - [ ] 출석부 관리 (이미지 → 데이터)
- [ ] Dify-n8n 연동 (HTTP Webhook)
- [ ] Conversation Opener 설정
- [ ] 에러 처리 및 자연어 응답
- [ ] 통합 테스트

**산출물**:
- 5개 Conversation Opener 버튼이 있는 AI Agent
- n8n 워크플로우 5개
- 통합 Docker Compose 패키지

**Agent 기능**:
```
✅ Phase 1 모든 기능
✅ [입금 대조] 버튼 → 자동 실행
✅ [계획서 검토] 버튼 → 자동 실행
✅ [수강자 목록] 버튼 → 자동 실행
✅ [보고서 생성] 버튼 → 자동 실행
✅ [출석부 업로드] → 이미지 OCR → 데이터 저장
✅ 실행 결과 자연어 설명
✅ 에러 상황 안내 및 해결 가이드
```

### 4.3 Phase 3: 능동적 업무 알림

**목표**: 일정 기반으로 필요한 업무를 능동적으로 안내

**기간**: Phase 2 완료 후 (3학기 중)

**범위**:
- [ ] 업무 일정 매핑 정의
- [ ] n8n Scheduler 워크플로우 개발
- [ ] Dify Agent UI에 알림 표시 연동
- [ ] 테스트 및 튜닝

**업무 일정 매핑 (예시)**:

| 시점 | 업무 | 알림 메시지 |
|-----|------|-----------|
| 학기 시작 2달 전 | 계획서 수합 | "계획서를 받아야 해요! 강사님들에게 강의 계획서를 받은 후 [드라이브 폴더]에 업로드해주세요" |
| 학기 시작 4주 전 | 홍보 게시 | "강의 홍보를 시작해야 해요!" |
| 수강신청 기간 | 입금 확인 | "입금 확인이 필요해요! [입금 대조] 버튼을 눌러주세요" |
| 개강 1주 후 | 환불 마감 | "환불 기간이 종료됩니다! 각 강좌별 환불 기간을 확인해주세요" |
| 학기 종료 후 | 보고서 작성 | "운영 보고서를 작성해야 해요!" |

**Agent 기능**:
```
✅ Phase 2 모든 기능
✅ 접속 시 현재 시점 기반 업무 안내 표시
✅ "지금 뭐 해야해?" 질문에 일정 기반 응답
```

## 5. 상세 설계

### 5.1 Dify Agent 설계

#### 5.1.1 System Prompt 구조

```markdown
# 역할
당신은 위례인생학교의 업무 도우미입니다.

# 컨텍스트
- 위례인생학교는 성인 대상 평생교육 프로그램입니다
- 1년 4학기제로 운영됩니다 (1월, 4월, 7월, 10월 시작)
- 주요 업무: 수강 등록 처리, 강의 계획서 관리, 수강자 목록 관리

# 제공 기능
1. 계획서 검토: 강사 제출 계획서 검토 및 게시글 생성
2. 입금 대조: 은행 입금 내역과 수강신청자 매칭
3. 출석부/수강자 목록: 강사별 수강자 목록 생성
4. 보고서 생성: 학기별/연도별 운영 현황

# 데이터 위치
- Google Drive: 위례인생학교 폴더
- 파일 구조: [별도 문서 참조]

# 행동 지침
1. 기능 요청 시: 해당 워크플로우 실행 → 결과 설명
2. 질문 시: 데이터 조회 또는 지식 기반 응답
3. 에러 발생 시: 원인 설명 + 해결 방법 안내
4. 범위 외 요청 시: 정중히 안내
```

#### 5.1.2 의도 분류 체계

| 의도 | 트리거 예시 | 처리 방법 |
|-----|------------|----------|
| `입금_대조` | "입금 대조", "입금 확인", "결제 확인" | n8n WF 호출 |
| `계획서_검토` | "계획서 검토", "강의계획서 확인" | n8n WF 호출 |
| `출석부_생성` | "출석부", "수강자 목록", "명단" | n8n WF 호출 |
| `보고서_생성` | "보고서", "현황", "통계" | n8n WF 호출 |
| `데이터_조회` | "~가 몇 명", "~를 알려줘" | Agent 직접 처리 |
| `일반_질문` | 기타 | Agent 직접 응답 |

### 5.2 n8n 워크플로우 설계 (Phase 2)

#### 5.2.1 입금 대조 워크플로우

```
[Webhook Trigger]
     │
     ├── Input: { year, semester }
     │
     ▼
[Google Drive: 입금내역 로드]
     │
     ▼
[Google Drive: 수강신청자 목록 로드]
     │
     ▼
[Code: 매칭 로직]
     │
     ├── 정상 매칭
     ├── 금액 기반 추정
     ├── 이름 불일치
     ├── 미입금
     └── 중복 입금
     │
     ▼
[Google Drive: 수강관리 파일 업데이트]
     │
     ▼
[Response]
     │
     └── Output: {
           status: "success" | "error",
           summary: { matched, pending, unmatched, duplicate },
           details: [...],
           error_message?: string
         }
```

#### 5.2.2 출석부 관리 워크플로우

```
[Webhook Trigger]
     │
     ├── Input: { image_base64, subject, date }
     │
     ▼
[Claude Vision: OCR]
     │
     ├── 이미지에서 출석 데이터 추출
     │   - 수강생 이름
     │   - 출석 여부 (O/X/지각 등)
     │
     ▼
[Code: 데이터 정리]
     │
     ▼
[Google Drive: 출석부 파일 업데이트]
     │
     ▼
[Response]
     │
     └── Output: {
           status: "success" | "error",
           subject: "경제해설(심화)",
           date: "2026-03-15",
           attendance: { present: 15, absent: 2, late: 1 },
           details: [...]
         }
```

#### 5.2.3 에러 응답 형식

```json
{
  "status": "error",
  "error_code": "FILE_NOT_FOUND",
  "error_message": "입금내역 파일을 찾을 수 없습니다",
  "details": {
    "expected_path": "위례인생학교/입금내역/2026_01_입금내역.xlsx",
    "suggestion": "Google Drive에 파일이 업로드되었는지 확인해주세요"
  }
}
```
