# 위례인생학교 업무 도우미 — 개발 기획서

> **문서 버전**: 5.0  
> **최종 수정**: 2026-03-09  
> **프로젝트명**: 위례인생학교 업무 도우미 (Wirye School Assistant)  
> **대상 독자**: 개발자, 프로젝트 관리자

---

## 1. 프로젝트 개요

### 1.1 목적과 가치

위례인생학교는 성인 대상 평생교육 프로그램으로, 연 4회차(1월/4월/7월/10월)에 걸쳐 다양한 강좌를 운영한다. 현재 2~4명의 관리자가 수강신청 접수, 입금 확인, 강의계획서 검토, 출석부 생성, 보고서 작성 등의 업무를 수작업으로 처리하고 있으며, 학기 시작 전 2~3주간 업무 부하가 집중된다.

이 프로젝트는 LLM과 Google Sheets API를 결합하여 관리자의 반복 업무를 자동화하되, **완벽한 자동화가 아니라 "수시간 → 수분"으로 단축하는 AI 보조 도구**를 목표로 한다. AI가 90%를 처리하고, 나머지 10%를 관리자가 빠르게 확인·수정하는 구조이다.

### 1.2 핵심 설계 원칙

- **프로토타입 우선**: 최소한의 뼈대(챗봇 + 배포)부터 만들고 점진적으로 기능을 추가한다.
- **완벽보다 속도**: AI 출력의 100% 정확성을 추구하지 않고, 관리자가 검증 가능한 구조화된 결과를 빠르게 제공한다.
- **LLM은 예외 처리 전담**: 코드로 처리 가능한 규칙 기반 로직은 코드로, LLM은 비정형 데이터 해석에만 사용하여 비용과 정확도를 최적화한다.
- **폴더/파일 명명 규칙이 곧 API**: 예측 가능한 파일 경로로 탐색 없이 직접 접근한다.

### 1.3 범위

**포함 (In-scope)**

- AI 기반 대화형 업무 도우미 (LangChain + Chainlit)
- Google Drive/Sheets 기반 데이터 저장/조회/수정
- 6개 핵심 업무 자동화: 입금 대조, 출석부 생성, 출석 체크(OCR), 계획서 검토, 보고서 생성, 질의응답
- 클라우드 배포 (Railway)

**제외 (Out-of-scope)**

- 배움숲 포털과의 직접 API 연동 (API 미제공)
- 수강생 대면 서비스 (관리자 전용 시스템)
- 모바일 전용 앱 개발
- 실시간 채팅/메시징, 일정 기반 능동적 알림 (추후 확장)

### 1.4 대상 사용자

위례인생학교 관리자 2~4명. IT 전문 인력이 아니며, 웹 브라우저에서 채팅 형태로 업무를 요청한다. 학기 시작 전 피크 시즌(연 4회, 각 2~3주)에 집중 사용하고, 평상시에는 간헐적으로 조회 용도로 사용한다.

## 2. 기술 스택

### 2.1 구성요소

| 구성요소 | 기술 | 용도 | 선택 근거 |
|---------|------|------|----------|
| LLM 프레임워크 | LangChain | LLM 호출 래퍼 | Python 네이티브, 모델 교체 용이 |
| 채팅 UI | Chainlit | 대화형 웹 인터페이스, 버튼 UI | Python 통합, Conversation Starter 버튼 네이티브 지원 |
| LLM | Claude API (Anthropic) | 텍스트 처리, OCR (Vision) | 한국어 성능 우수, Vision 지원 |
| 데이터 저장 | Google Drive / Sheets API | 모든 업무 데이터 저장/관리 | 관리자가 직접 확인/수정 가능, 별도 DB 불필요 |
| Google API 인증 | Service Account + Domain-wide Delegation | Drive/Sheets 읽기/쓰기 | SA가 Workspace 사용자 대리로 파일 생성 가능 |
| 배포 | Railway | 운영 환경 | Git push 자동 배포, HTTPS/도메인 자동, 관리 부담 최소 |
| 프로그래밍 언어 | Python 3.10+ | 전체 앱 | LangChain/Chainlit 네이티브 |

### 2.2 아키텍처 방침

**고정 파이프라인 + LLM은 특정 단계에서만 (Agent 패턴 사용 금지)**

LangChain Agent나 tool-calling agent 패턴을 사용하지 않는다. 각 작업은 실행 순서가 고정된 Python 함수 파이프라인으로 구현하고, LLM은 비정형 텍스트 해석이 필요한 단계에서만 호출한다. 의도 분류는 Conversation Starter 버튼의 고정 메시지로 판별한다.

LangChain Skills(`.agents/skills/`)는 Claude Code가 코드 작성 시 참조하는 가이드이며, 런타임 동작에는 영향 없다.

## 3. 비즈니스 컨텍스트

### 3.1 회차 구조

연 4회차, 각 회차 3개월, 강좌당 최대 12회차.

| 회차 | 계절 | 시작월 | 기간 |
|:---:|:----:|:-----:|:----:|
| 1 | 겨울 | 1월 | 1~3월 |
| 2 | 봄 | 4월 | 4~6월 |
| 3 | 여름 | 7월 | 7~9월 |
| 4 | 가을 | 10월 | 10~12월 |

회차 식별자: `{연도}-{번호}` (예: `2026-1`)
회차명: `{연도}-{번호} {계절}학기` (예: `2026-1 겨울학기`)

### 3.2 회원 등급 체계

3단계 등급 구조. 승급/강등이 회차 흐름에 따라 자동 또는 입금 기반으로 전환된다.

```
[비회원] ──가입비──→ [회원] ──수강료 입금──→ [준회원] ──종강──→ [회원]
                       │
                       └──정회원비──→ [정회원] ──후년1학기(겨울) 종강──→ [회원]
```

| 등급 | 승급 조건 | 수강 신청 | 수강료 | 유효기간 | 만료 시 |
|------|---------|:--------:|:------:|---------|--------|
| 회원 | 가입비 납부 | 가능 | 필요 | 영속 | — |
| 준회원 | 수강료 입금 완료 (1과목 이상) | 등록 완료 | 납부 완료 | 해당 회차 종강까지 | 회원으로 강등 |
| 정회원 | 정회원비 납부 | 가능 | **면제** | ~후년 1학기(겨울) 종강까지 | 회원으로 강등 |

비회원은 시스템 외부 상태이며, 가입 완료 시 회원관리 시트에 추가된다.

**특수 케이스**:
- 강사: 자동으로 정회원 자격 부여 (시작/만료 기준은 추후 확정). 1학기 종강 시 일괄 강등에서 제외.
- 사무처 직원: 입사 시 정회원, 퇴사 시 해당 회차까지 유지. 1학기 종강 시 일괄 강등에서 제외.

### 3.3 비용 구조

| 항목 | 기본 금액 | 비고 |
|------|:--------:|------|
| 가입비 | 1만원 | 변동 가능 |
| 수강료 | 2만원 (전 과목 동일) | 변동 가능 |
| 정회원비 | 12만원 | 매년 초(약 2월) 결정, 변동 |

모든 비용은 변동 가능하므로 코드 하드코딩이 아닌 설정값으로 관리한다.

### 3.4 환불

- 환불 마감: 개강 후 7일, 이후 환불 불가

### 3.5 사용자 식별

- **이름ID**: `이름+핸드폰번호뒷자리` (예: `박민서6804`)
- 동명이인 구분 목적
- 마스터 시트 및 출석부의 기본 키

### 3.6 주요 업무 일정

| 시점 | 업무 |
|-----|------|
| 학기 시작 2달 전 | 강사에게 강의 계획서 요청 |
| 학기 시작 1달 전 | 계획서 검토, 배움숲/카페 홍보 게시 |
| 전월 말일까지 | 수강신청 접수 마감 |
| 개강 첫 주 전 토요일 | 강사에게 수강자 목록(출석부) 전달 |
| 개강 1주 후 | 환불 마감 |
| 학기 종료 후 | 출석 체크(OCR), 운영 보고서 작성, 종강 처리 |

## 4. 데이터 구조

### 4.1 Google Drive 폴더 구조

```
위례인생학교 자료실/
├── 회원관리 (Google Sheets)           ← 전체 회원 마스터 (영속)
├── 수강기록 (Google Sheets)           ← 전체 수강 이력 (영속)
└── 학사운영(연도별)/
    └── {연도}/
        └── {회차} {계절}학기/          ← 예: "2026-1 겨울학기"
            ├── LEARNING_APPLY*.xls    ← 배움숲 신청자 목록 (SoT)
            ├── 수강생 (Google Sheets)  ← Agent가 신청자 목록 기반으로 생성
            ├── 출석부/
            │   ├── 출석부 (Google Sheets)  ← 과목별 시트탭
            │   └── {회차}_{과목명}_출석부.pdf ← A4 프린트용
            └── 강의계획서/
                ├── 초안/
                └── 검토완료/
```

루트 레벨에 영속 데이터(회원관리, 수강기록), 회차 폴더에 회차별 데이터. 파일명/폴더명에 회차 정보가 포함되어 AI가 경로를 직접 구성할 수 있다.

### 4.2 테이블 관계

```
[회원관리] ←──집계── [수강기록]
 (루트, 영속)          (루트, 영속)
     │                    ▲
     │ 이름ID             │ 종강 시 추가
     ▼                    │
[수강생]              [출석부]
 (회차 폴더)           (회차 폴더)
```

### 4.3 회원관리 (루트, 영속)

전체 회원(과거+현재)을 관리하는 마스터 테이블. 신규 가입 시 행 추가, 회차가 넘어가도 유지.

| 컬럼 | 설명 | 비고 |
|------|------|------|
| 이름ID | 이름+뒷자리 (예: 박민서6804) | PK |
| 이름 | | |
| 성별 | | |
| 전화번호 | 010-XXXX-XXXX | |
| 주소 | | |
| 나이 | | |
| 등급 | 회원 / 준회원 / 정회원 | 승급/강등 자동 관리 |
| 가입날짜 | | |
| 수강count | 누적 수강 횟수 | 수강기록에서 집계 |
| 출석률(누적) | 전체 평균 출석률 | 수강기록에서 집계 |
| 마지막수강학기 | | 수강기록에서 집계 |

**등급 관리 로직**:
- **회원**: 가입비 납부 완료. 수강 신청 가능. 영속(만료 없음).
- **준회원**: 수강료 입금 확인 시 회원에서 승급 (1과목 이상). 종강 후 회원으로 자동 강등.
- **정회원**: 정회원비 납부 시 회원에서 승급. 수강료 면제. 후년 1학기(겨울) 종강 시 회원으로 강등.
- 강사, 사무처 직원: 정회원 일괄 강등 시 예외 처리.

**자동 등급 전환 시점**:
- 입금 대조 완료 후: 수강료 입금 확인 → 등급 "회원" → "준회원" 승급
- 종강 시: 준회원 → "회원" 강등
- 1학기(겨울) 종강 시: 정회원 → "회원" 일괄 강등 (강사/사무처 예외)

### 4.4 수강기록 (루트, 영속)

전체 수강 이력을 관리하는 테이블. 종강 시 출석부 데이터를 바탕으로 행을 추가한다.

| 컬럼 | 설명 |
|------|------|
| 이름ID | 회원관리와 연결 |
| 회차 | 2026-1 등 |
| 과목명 | |
| 출석률 | 해당 강좌의 출석률 |

종강 시 흐름: 출석부 → 출석률 확정 → 수강기록에 행 추가 → 회원관리의 수강count/출석률/마지막수강학기 갱신.

### 4.5 수강생 (회차 폴더, 회차별)

해당 회차의 수강 신청자와 입금 상태를 관리하는 테이블. 회차마다 Agent가 신청자 목록 기반으로 새로 생성. 다과목 수강자는 이름ID+과목명이 복합 PK로, 과목마다 행이 추가된다.

| 컬럼 | 설명 | 입력 주체 |
|------|------|----------|
| 이름ID | 회원관리와 연결 | **Agent** (신청자 목록에서 생성) |
| 과목명 | | **Agent** |
| 입금시간 | 매칭된 입금 건의 거래일시 | **Agent** (입금 대조 시) |
| 입금자명(적요) | 매칭된 입금 건의 은행 적요 (원본) | **Agent** |
| 비고 | | |
| 입금현황 | ✅정상 / 🔶확인필요 / ⚠️이름불일치 / ❌미입금 / 🔄중복 / 💎면제 | **Agent** |
| 등록상태 | 정상등록 / 대기 / 취소 | 관리자 확인 후 최종 결정 |

**생성 원본**: 배움숲 신청자 목록 (`LEARNING_APPLY*.xls`)에서 이름ID, 과목명을 추출.

### 4.6 출석부 (회차/출석부/ 폴더)

Google Sheets 1개 파일, 과목별 시트탭.

| 컬럼 | 설명 |
|------|------|
| ID | 이름+뒷자리 |
| 이름 | |
| 1회차 ~ 12회차 | 출석/결석/지각 등 (12회차 일괄 생성) |
| 출석률 | 출석수 / 회차수 × 100 (수식 삽입) |

등록상태가 정상인 수강자만 포함. 과목별 PDF도 생성 (A4 프린트용, 파일명: `{회차}_{과목명}_출석부.pdf`).

### 4.7 신청자 목록 (배움숲 다운로드 원본)

관리자가 배움숲에서 다운로드하여 회차 폴더에 업로드하는 엑셀 파일. 수강생 시트 생성의 SoT(Source of Truth).

**파일명 패턴**: `LEARNING_APPLY{datetime}.xls` (HTML 형식 .xls)

**파싱 주의사항**: 23개 헤더 vs 22개 데이터 셀 — `실제결제금액` 컬럼이 헤더에만 존재하고 데이터가 없어 1칸씩 밀림. BeautifulSoup으로 HTML 파싱 후 보정 필요.

**수강생 시트에 필요한 정보**: 강좌명, 신청자(아이디) — 여기서 이름과 전화번호를 추출하여 이름ID 생성. 추가 정보(성별, 나이, 주소 등)는 회원관리 시트에서 관리.

**신청상태**: 결제완료, 결제대기, 접수완료, 취소, 환불완료, 환불신청, 기간만료, 결제취소 — 수강생 시트 생성 시 전체 포함 (취소/환불은 관리자 판단).

### 4.8 입금내역 (은행 엑셀)

관리자가 챗봇에 직접 업로드 (Drive에 별도 저장하지 않음).

| 컬럼 | 설명 | 비고 |
|------|------|------|
| 거래일시 | 2025-12-31 23:42:54 | |
| 적요 | 비정형 입금자 입력 텍스트 | 이름+강좌 혼합, 형식 자유, 매칭 핵심 |
| 의뢰인/수취인 | 은행 계좌 명의자 | 이름 매칭 핵심 |
| 입금 | 금액 | |
| 출금 | | |
| 거래후잔액 | | |
| 구분 | 대체/타행이체 | |
| 거래점 | 은행명(계좌뒷자리) | |

헤더가 7행에 위치하고, 1~6행은 계좌 메타정보, 마지막 행은 합계. 비고 컬럼은 없는 것으로 취급.

### 4.9 입금자명 패턴 (실제 데이터 기반)

은행 적요 컬럼은 입금자가 자유롭게 입력하는 텍스트로, 형식이 통일되어 있지 않다.

| 패턴 | 예시 | 빈도 |
|------|------|------|
| 이름+강좌(구분자 없음) | `김기춘경제뉴스로기초` | 매우 많음 |
| 이름_강좌 | `류동원_요들` | 일부 |

| 이름-강좌, 이름/강좌, 이름.강좌 | `김혜경-심리상담`, `조현진/금융과경제` | 일부 |

## 5. 핵심 기능

### 5.1 기능 구성

| # | 작업 | 우선순위 | 설명 |
|---|------|:-------:|------|
| 1 | **입금 대조** | 🔴 1순위 | 신청자 목록 → 수강생 시트 생성 → 은행 입금내역 매칭 |
| 2 | **출석부 생성** | 🔴 1순위 | 수강생 시트에서 등록 완료자 추출 → 강좌별 출석부 생성 |
| 3 | **출석 체크 (OCR)** | 🟡 2순위 | 종이 출석부 사진 → Vision OCR → 출석부 시트 업데이트 |
| 4 | **계획서 검토** | 🟡 2순위 | PDF 파싱 → 오탈자/말투 수정 → 배움숲 게시 멘트 생성 |
| 5 | **종강 처리** | 🟡 2순위 | 출석률 계산, 수강기록 추가, 등급 강등, 보고서 |
| 6 | **보고서 생성** | 🟢 3순위 | 회차별 운영 현황 보고서 자동 생성 |
| 7 | **질의 응답** | 상시 | 업무 규칙, 일정, 회원 제도 등 Q&A |

### 5.2 입금 대조 (1순위) — 전체 플로우

AI 효율화의 핵심 작업. 신청자 목록 기반 수강생 시트 생성 + 코드 기반 규칙 매칭(80~90%) + LLM 기반 예외 처리(10~20%).

**플로우** (관리자 ↔ Agent 상호작용):

1. **관리자**: 배움숲에서 신청자 목록 엑셀 다운로드 → Drive 회차 폴더에 업로드
2. **관리자**: 챗봇에서 "💰 입금 대조" Starter 버튼 클릭
3. **Agent**: 현재 날짜 기반 회차 추측 → "2026-1 겨울학기 입금 대조를 시작할까요?" → 관리자 확정/수정
4. **Agent**: Drive에서 해당 회차 폴더의 `LEARNING_APPLY*.xls` 찾기 → 파싱 → 수강생 Google Sheets 생성
5. **Agent**: "수강 신청자(총 N명)를 확인했습니다. 입금 내역을 업로드해주세요"
6. **관리자**: 입금 내역 엑셀을 챗봇에 직접 업로드
7. **Agent**: 회원관리 + 수강생 + 입금내역 바탕으로 매칭 실행 → 수강생 시트 업데이트
8. **Agent**: 결과 요약 + 시트 링크 + "배움숲에서 결제완료 회원의 등록상태를 변경해주세요"
9. **Agent**: Action 버튼 — 📋 출석부 생성, 🔄 입금대조 다시하기, ❓ 다른 질문하기
10. **관리자**: 수동 확인 후 '출석부 생성' 클릭
11. **Agent**: 출석부 생성 (5.3절) + 회원관리 업데이트 + 수강기록 추가

**매칭 로직**:

정회원 선처리 → 코드 매칭 → LLM 매칭 (미매칭 건만) → 결과 시트 기록

코드 매칭 순서:
1. 비수강료 필터링: 금액 < 1만원 스킵, "취소"/"대기" 키워드 감지
2. 이름 매칭: 의뢰인 컬럼 → 수강생 이름 (정확 일치). 카카오페이/토스면 적요에서 추출
3. 강좌 매칭: 적요 컬럼에서 강좌 키워드 추출 → 과목명 대조
4. 금액 분류: 1만(가입비), 2만(수강료), 3만(합산), 4만+(다과목), 12만(정회원)
5. 동명이인: 이름 매칭 2명+ → 강좌명으로 2차 구분

**입금 상태 코드** (6가지):

| 코드 | 이모지 | 의미 |
|-----|:-----:|------|
| confirmed | ✅ | 정상 매칭 |
| needs_review | 🔶 | 확인 필요 |
| name_mismatch | ⚠️ | 이름 불일치 (대리 입금 추정) |
| not_paid | ❌ | 미입금 |
| duplicate | 🔄 | 중복 입금 |
| exempted | 💎 | 정회원 — 수강료 면제 |

### 5.3 출석부 생성 (1순위)

입금 대조 완료 후의 후속 단계. 별도 Starter 버튼으로도 독립 실행 가능.

**처리 개요**:
1. 수강생 시트에서 등록상태 "정상등록" 수강자 필터링
2. 강좌별 그룹핑 → 중간 확인 ("금융과 경제: 17명, ...")
3. 회차/출석부/ 폴더에 '출석부' Google Sheets 생성 (과목별 시트탭)
4. 각 시트: ID, 이름, 1~12회차, 출석률 수식
5. 과목별 PDF 생성 (A4 프린트용, 파일명: `{회차}_{과목명}_출석부.pdf`)
6. 회원관리 업데이트: 수강등록 완료된 회원→준회원 승격, 마지막수강학기 수정, 수강count +1
7. 수강기록에 행 추가

### 5.4 종강 처리 (2순위, 백로그)

회차 종료 후 일괄 처리. 상세 설계는 추후 확정.

**처리 개요**:
1. 출석부의 과목 시트별 수강생별 출석률 계산
2. 수강기록 시트에 출석률 추가
3. 회원관리 시트의 누적 출석률 갱신
4. 준회원 → 회원 일괄 강등
5. 1학기(겨울) 종강 시: 정회원 → 회원 일괄 강등 (강사/사무처 직원 예외)
6. 운영 보고서 생성
7. 추가 필요 작업 확인 예정

### 5.5 출석 체크 — OCR (2순위)

종강 후 강좌별 1회 진행. 실물 출석부(12회차 전체)를 사진으로 촬영하여 디지털화.

### 5.6 계획서 검토 (2순위)

**지원 형식**: PDF만 (HWP 미지원)

**처리 우선순위**:
- 1순위: 오탈자 수정, 말투 수정, 배움숲 업로드용 카테고리별 복붙 멘트 생성
- 후순위: 내용 보강, 용어 보완, 네이버 카페 멘트 생성

### 5.7 질의 응답

비즈니스 규칙에 대한 자연어 Q&A. 시스템 프롬프트에 비즈니스 컨텍스트를 직접 주입하여 처리한다.

```python
credentials = service_account.Credentials.from_service_account_file(

)
```

PaaS 배포 시: `GOOGLE_SA_KEY_JSON` 환경변수 → `from_service_account_info()` 사용.
